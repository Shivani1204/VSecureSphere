Procedure

Step 1: Start Snort on Victim System

On the victim container, start Snort with appropriate configurations:

            sudo snort -A console -i eth0 -c /etc/snort/snort.conf

This command starts Snort in console mode, monitors the specified interface (e.g., eth0), and uses the default configuration file.

Step 2: Generate Traffic from Attacker

From the attacker container, simulate suspicious traffic using:

            sudo hping3 -S <victim-ip> -p 80 --flood

Or use Nmap to simulate a port scan:

            sudo nmap -sS <victim-ip>

Step 3: Observe Snort Alerts

In the victim terminal where Snort is running, observe the console output for alerts triggered by the simulated traffic. These might include:

  • Port scan detected
  • SYN flood attempt
  • TCP/UDP traffic anomalies

Step 4: Analyze the Logs

Snort may also generate logs (if configured) in:

            /var/log/snort/

Use cat, less, or grep to analyze these logs and identify the pattern of intrusion.

Step 5: Discuss Mitigation and Response

Discuss how IDS systems like Snort help detect network-based attacks. Points to consider:

  • Types of traffic detected and flagged by Snort
  • Role of rule configuration in accuracy of alerts
  • Use of IDS in real-world enterprise setups for early threat detection

Step 6: Conclude the Experiment

Summarize the importance of IDS in monitoring and defending network environments. Emphasize how proper traffic inspection and alerting mechanisms can reduce the risk of breaches.
